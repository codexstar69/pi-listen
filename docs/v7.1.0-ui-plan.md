# pi-listen v7.1.0 — Settings UI redesign (v5 — final lifecycle hardening)

## Mission

Take the Speak tab + pickers from "functional" to "world-class terminal
UI" while fixing the architectural gaps Codex and Gemini flagged in v1.
The visual goals are unchanged; v2/v3/v4 add the missing ownership
model, cancellation contract, frame coalescer, honest playback indicator,
and the lifecycle hardening (unregister, ticker ownership, dispose
isolation, per-instance install keys) that Codex and Gemini called out
on the v3 pass.

## Hard constraints (unchanged from v1)

- No emoji. Geometric Unicode (U+2500-25FF) and a small allowlist of
  utility marks (☐ ☑ ◐ ◓ ◑ ◒ ● ○ → ← ↑ ↓ ⇒ ›) only.
- Theme-aware everywhere via `theme.fg(role)`.
- One picker chassis shared by Lang / Models / Voice — defined in §3.
- Backwards compatible: zero v7.0.x slash command / config / keybinding
  removals.
- No new third-party deps.
- Fits ≥60-column terminals; degrades cleanly below (see §10).
- Render under 16ms / animation timers ≤10 Hz / coalesced (see §2).
- A11y: every signal has a non-color complement.

## §1 — Widget ownership model (NEW; addresses Codex #1, Gemini critical)

### Problem

`ctx.ui.setWidget(key, content)` writes to a key-slotted store. Two
widgets with different keys coexist; two widgets with the same key
overwrite. Today voice.ts uses `"voice-recording"` for both warmup AND
recording, so the recording widget cleanly replaces the warmup widget
when the state machine advances. v7.1 introduces THREE new widget keys
that must coexist with `voice-recording`:

- `installWidgetKey(modelId)` (e.g. `voice-tts-install:kitten-nano-en-v0_2`)
  — sticky download progress, **per-model-id slot** so concurrent
  installs of different models coexist
- `voice-tts-playback` — playback indicator
- (no key for help / onboarding overlays — they use `ctx.ui.custom()`
  which is overlay, not a widget slot)

### Rules

1. **Each widget owns exactly ONE key.** No widget reuses another's key.
2. **Each widget exposes `dispose()` that is idempotent.** Calling it
   twice is a no-op. Internally, the order is (Codex v4 #2 — set
   `disposed = true` BEFORE any cleanup work so reentrant onProgress
   / ticker callbacks running mid-teardown short-circuit cleanly,
   while still checking prior state first to make the idempotency
   guard correct — Codex v5 nit):
   (a) checks `if (this.disposed) return;` (idempotency guard reads
       PRIOR state so re-entry is a no-op),
   (b) sets `this.disposed = true` BEFORE any further cleanup,
   (c) calls the ticker-unsubscribe handle returned by
       `ticker.subscribe(...)` so the ticker's refcount drops and its
       `setInterval` can stop when no widgets remain (Codex v4 #1),
   (d) clears any non-ticker animation timer it owns,
   (e) calls `ctx.ui.setWidget(key, undefined)` to clear the slot,
   (f) calls `registry.unregister(this.key, this)` so the registry
       evicts this widget — owner-checked, see §1 unregister contract.
3. **`voiceCleanup()` is the single sink.** Registered widgets append
   their `dispose()` to a `cleanupTasks: (() => void)[]` array. On
   session_shutdown / session_start (non-startup) / disable, the array
   is drained in registration order, idempotently.
4. **Late progress callbacks are no-ops.** Every widget guards its
   onProgress handler with `if (this.disposed) return;` so a fetched
   tar chunk landing 200ms after dispose can't re-render the slot.
5. **Conflict policy**: simultaneous install + recording is allowed
   (both widgets stack as belowEditor placement; user sees BOTH).
   Simultaneous install + playback meter is also allowed. The plan
   does NOT use a single global slot.

### API contract

```ts
// new file: voice/ui-widget-base.ts
export interface DisposableWidget {
    /** Stable widget-key the slot writes to. Install uses
     *  `installWidgetKey(modelId)`; recording uses "voice-recording";
     *  playback uses "voice-tts-playback". */
    readonly key: string;
    dispose(): void;
}

export interface WidgetRegistry {
    /**
     * Register a widget by its `key`. If a widget with the same key is
     * already registered, dispose() is called on it FIRST (synchronously),
     * then the new widget takes the slot. This prevents same-key
     * collision: e.g. two concurrent `tts-install-progress` widgets for
     * the same model id would otherwise have the second's `dispose()`
     * wipe the first's slot. The registry serializes ownership.
     */
    register(w: DisposableWidget): void;
    /**
     * Owner-checked eviction (Codex v4 #3). Removes the entry for
     * `key` from the internal Map ONLY IF the current entry is
     * `owner` (identity-equal). If a stale dispose path runs after
     * a NEW widget has taken the same key (e.g. an old install
     * widget's queued cleanup firing after `register()` swapped in a
     * fresh one), the unregister is a no-op and the new widget
     * stays bound. Without this guard, a late dispose could evict
     * its successor.
     *
     * Also addresses Gemini v3 #1 — memory leak. Called by a
     * widget's own `dispose()` after it tears down its slot, so
     * completed widgets do not pile up in the Map across long
     * sessions (e.g. a user installing 50 different models).
     * Idempotent: calling `unregister(key, owner)` when `key` is
     * absent OR points to a different owner is a no-op.
     */
    unregister(key: string, owner: DisposableWidget): void;
    /**
     * Drains the registry. Implementation iterates over a CLONED
     * snapshot of the Map's values (Gemini v4 implementation note),
     * because each `dispose()` synchronously calls
     * `unregister(this.key, this)` which mutates the underlying
     * Map. Cloning ensures no widget is skipped during the
     * teardown cascade. Each `dispose()` call is wrapped in
     * `try/catch` (Gemini v3 #2) so a single throwing widget
     * cannot abort cleanup for the rest. Idempotent.
     */
    disposeAll(): void;
}

/**
 * Per-instance install widget key. Different model ids occupy
 * different slots so concurrent installs (kitten + kokoro) coexist
 * without clobbering each other. Codex v3 #1.
 */
export function installWidgetKey(modelId: string): string {
    return `voice-tts-install:${modelId}`;
}

export function makeWidgetRegistry(): WidgetRegistry { … }
```

**Same-key collision rule (addresses Gemini v2 finding #1):** the
registry's internal store is `Map<string, DisposableWidget>`, NOT an
array. `register(w)` calls `existing.dispose()` synchronously before
swapping in `w`. This means the higher-level orchestrator (e.g.
`runSpeak` triggering an install widget) is responsible for serializing
installs OR explicitly accepting that the second install wins the
slot. In practice, the v7.0.1 in-flight Map in
`ensureTtsModelInstalled` ALREADY serializes same-id installs into one
promise, so two visible install widgets for the same id can't happen.
Different-id installs run in different keys (e.g.
`voice-tts-install:kitten-nano-en-v0_2` vs
`voice-tts-install:kokoro-int8-multi-lang-v1_0`) and coexist without
collision — the install widget's key encodes the model id.

`voice.ts` instantiates one registry per session (in `session_start`),
adds new widgets via `registry.register(...)`, and calls
`registry.disposeAll()` in `voiceCleanup`. Two-line addition.

## §2 — Frame coalescer (NEW; addresses Gemini "drift/tearing")

### Problem

Three independent `setInterval(…, 100ms)` widgets drift relative to each
other and each call `ctx.ui.setWidget()` separately, causing 20-30 partial
re-renders/sec instead of 10 cohesive ones.

### Solution

A single shared `RenderTicker` running at exactly 10 Hz (100ms
setInterval). Widgets subscribe via `ticker.subscribe(callback)`; the
ticker invokes all callbacks in a single tick, batching their
`setWidget` calls within the same JS turn. The Pi TUI then sees ONE
batch of slot updates per tick instead of three separate ones.

```ts
// new file: voice/ui-render-ticker.ts
export interface TickerSubscriber {
    /** Called once per tick (10 Hz). May throw; ticker isolates. */
    readonly tick: () => void;
    /**
     * Optional ownership hook. If present and the subscriber is auto-
     * unsubscribed after repeated throws, the ticker invokes
     * `dispose()` to tear down the broken widget's slot. The ticker
     * does NOT assume ownership when `dispose` is omitted (e.g. ad-hoc
     * test subscribers).
     */
    readonly dispose?: () => void;
    /** Optional debug label, shown in voiceDebug() output. */
    readonly label?: string;
}

export interface RenderTicker {
    /** Subscribe an owned subscriber. Returns an unsubscribe fn. */
    subscribe(subscriber: TickerSubscriber): () => void;
    refCount(): number;                      // for tests / debugging
    dispose(): void;
}

export function makeRenderTicker(): RenderTicker { … }
```

**Subscriber failure isolation (addresses Gemini v2 finding #2):** the
ticker iterates subscribers with `try { sub.tick(); } catch (err) { … }`
per subscriber so a single throwing widget cannot crash the loop and
freeze the rest. Errors caught are logged via `voiceDebug()` (a no-op
when `PI_VOICE_DEBUG` is unset, so production users see nothing) but
the ticker keeps ticking.

**Eviction with ownership (addresses Codex v3 #2 + Gemini v3 #2):** a
subscriber that throws 3 consecutive ticks is auto-unsubscribed. The
ticker then calls `sub.dispose?.()` to tear down the broken widget's
slot — but only because the subscriber was passed as an object
explicitly carrying its `dispose` reference, so ownership is explicit,
not inferred. The eviction call itself is wrapped in `try/catch`: if a
broken widget's `dispose()` throws (because its internal state is
corrupted enough to throw on render, the same corruption can crash
dispose), the failure is logged via `voiceDebug()` and the ticker
loop survives to keep the other widgets' frames flowing.

The ticker's `setInterval` is started lazily on first subscriber and
stopped on last unsubscribe (no idle-tick CPU when no widgets need
updates). Same singleton scope as the widget registry.

## §3 — Picker chassis contract (NEW; addresses Codex #5)

### Defined behaviors for ALL pickers (Lang, Model, Voice)

1. **Headings are non-selectable.** ↑↓ skips them via the same
   `selectableIdx[]` indirection the v5 STT Models picker already uses.
2. **Search filtering also skips headings.** A heading appears iff at
   least one row under it matches the filter; otherwise drop the entire
   group + heading from the view.
3. **Cursor restoration**: when search query changes, cursor moves to
   the first selectable row. When search clears, cursor moves to the
   previously-active row if visible, else first selectable row.
4. **Empty state**: if no rows match, render "No matches for '<query>'"
   in dim text. Footer shows `[esc] back  [bksp] clear search`.
5. **Page bounds**: cursor wraps top-to-bottom and vice versa within
   selectable rows only.
6. **Width fallback**: if available width < 60 columns, picker switches
   to compact mode — drop heading rows, drop install-status badges,
   keep just `name + size + active marker`. See §10.

### Code reuse

A new `PickerChassis<T>` class in `voice/ui-picker.ts` encapsulates the
state machine. The three concrete pickers (Lang/Model/Voice) extend or
compose it. The current settings-panel.ts duplicates this logic three
times — v7.1 unifies it.

## §4 — Cancellation contract (REVISED; addresses Codex #2)

### Problem with v1

v1 said "the widget aborts via the signal." `AbortSignal` is read-only;
you abort an `AbortController`. v1 also didn't define ordering of
[esc] when multiple things might consume it (panel, picker, install
widget, help overlay).

### v2 contract

1. **Install widget receives an `AbortController`** (not signal).
   Pressing [esc] in the install widget calls `controller.abort()`.
   The widget's onProgress handler also stops emitting after abort.
2. **Escape priority order**, top-to-bottom:
   - help overlay open → close help
   - voice picker open → back to model picker
   - model picker open → back to main panel
   - install widget mounted → cancel install (abort controller)
   - main panel open → close panel
3. **Panel-close-vs-widget-cancel**: closing the panel does NOT cancel
   in-flight installs. Install widget remains visible (belowEditor)
   and continues until done or user explicitly cancels via [esc] when
   no overlay is in front of it.
4. **Idempotent cancellation**: aborting a controller after the install
   already finished is a no-op. The post-install promise has already
   resolved; the abort signal handler unsubscribes itself in
   `finally`. Verified in tests.

## §5 — Pillar A: Install progress widget (REVISED)

Same visual as v1 (`█` filled / `░` empty bar, speed, ETA), but now:

- **Per-instance widget key** via `installWidgetKey(modelId)` (Codex
  v3 #1) — concurrent installs for different model ids occupy
  different slots and coexist; same-id installs are already serialized
  by the v7.0.1 in-flight Map in `ensureTtsModelInstalled`.
- Receives an `AbortController` in its constructor (not signal).
- Implements `DisposableWidget` from §1.
- On construction, calls
  `this.unsubTicker = ticker.subscribe({ tick: () => this.onTick(), dispose: () => this.dispose(), label: \`install:${modelId}\` })`
  and stores the returned unsubscribe handle on `this`.
- onProgress callback is guarded by `disposed` flag.
- All cleanup paths idempotent.
- Speed/ETA computed from byte-counter snapshots taken during ticker
  callbacks (not directly in onProgress) — keeps ticker as single
  authoritative animation source.
- On `phase: "done"` it calls `this.dispose()` exactly once. The §1
  dispose ordering then runs: set `disposed = true` → call
  `this.unsubTicker?.()` so the ticker's refcount drops (Codex v4 #1
  — without this, a completed widget keeps the ticker alive until a
  failure-eviction or session teardown) → clear the slot via
  `setWidget(key, undefined)` → `registry.unregister(this.key, this)`
  with owner-checked semantics (Gemini v3 #1 + Codex v4 #3).

## §8 — Pillar D: Native-script labels (REVISED; addresses Codex #6 + Gemini high)

### Width-aware rendering

Use the existing `truncateToWidth` from `pi-tui` PLUS a new helper
`visualWidth(str)` that handles surrogate pairs and East-Asian wide
characters (uses Unicode 15.1 EAW Wide / Fullwidth properties — small
table, ~50 LOC, hand-curated to the scripts we ship).

Each voice picker row computes:
- Romanized name (always shown, ≤16 chars left-aligned)
- Native-script label (right-aligned, only if remaining width ≥
  visualWidth(label) + 2)

### RTL / Arabic handling

Arabic voices (`piper-ar_JO-kareem`) get the romanized label only —
RTL embedding in fixed-width LTR terminal layouts breaks cursor
positioning. The Arabic name `العربية` could be rendered, but the
risk-of-misrendering-on-some-terminals outweighs the discoverability
gain. Document this in the help overlay.

### Static map shape

```ts
// voice/ui-locale-labels.ts
interface LocaleLabel {
    nativeName: string;       // "中文" | "日本語" | "한국어"
    masc?: string;            // "男性" | "男性" | "남성"
    fem?: string;             // "女性" | "女性" | "여성"
}
const LOCALE_LABELS: Record<string, LocaleLabel> = { … };
// Hand-curated: en, zh, ja, ko, es, fr, de, it, pt, ru, nl, tr.
// ar intentionally omitted — see RTL handling.
// hi (Hindi) intentionally omitted — Devanagari uses combining marks
// and zero-width vowels (ZWJ/ZWNJ) that a simple width calculator
// (length + EAW table) overcounts. Without a real grapheme segmenter
// (would violate the zero-dependency constraint) we'd render mis-
// aligned right-aligned columns and break truncation. Hindi voices
// get the romanized fallback `(Hindi · M/F)` only.
```

Unknown base languages fall back to English `(language · gender)`. No
guessing. The omissions list (`ar`, `hi`) is documented in the help
overlay so users know why their language doesn't get native script.

## §9 — Pillar E: Onboarding overlay (REVISED; addresses Codex #7)

### Defined event ordering

User presses [enter] in the onboarding overlay:

1. Mark `ttsOnboardingShown = true` and persist config IMMEDIATELY
   (before any async work). User won't see the hint again even if
   the rest fails.
2. Call `runSpeak(ctx, "The quick brown fox …", { forceEnabled: true })`.
3. `runSpeak` triggers auto-install via the new install widget (§5)
   if model not present.
4. If install/play fails: surface the error via `notify`, leave config
   in whatever state user picked.
5. If user presses [esc] during install (cancel): same as (4) but
   error is "Onboarding cancelled — run /voice-speak-test when ready."

User presses [m] (pick a different model):

1. Mark `ttsOnboardingShown = true` (same idempotency).
2. Close onboarding overlay.
3. Open the Speak tab Models picker directly via the panel-action mechanism.

User presses [esc] (skip):

1. Mark `ttsOnboardingShown = true`.
2. Close onboarding.

## §10 — Pillar H: Width handling (REVISED; addresses Codex #9)

### Three width tiers

| Width | Behavior |
|---|---|
| ≥80 cols | Full chrome — borders, two-row status, install bar with speed+ETA, native-script labels in voice picker |
| 60-79 cols | Trimmed chrome — single-row status, install bar without ETA (just %), native-script labels truncated to base-language column |
| <60 cols | Hard block — panel renders a single-line "terminal too narrow — resize to ≥60 cols" message and disables interaction. Slash commands still work non-visually (`/voice-speak`, `/voice-speak-info`, `/voice-speak-models` opens a notify with text instructions, etc.) |

**Width-tier contradiction resolved (addresses Gemini v2 finding #4):**
v2 had two conflicting descriptions of the <60-col tier — "minimal
functional UI" AND "hard block." v3 picks ONE: hard block. Reasons:
(1) a functional UI below 60 cols would force per-component reflow
logic that doubles the picker chassis complexity; (2) the panel is
rarely opened on terminals that narrow — it's a settings overlay, not
an always-on widget; (3) the slash-command surface (`/voice-speak`,
`/voice-speak-info`, `/voice-speak-models`) provides text-based access
that doesn't need the panel chrome.

`ctx.ui.custom()` overlay options updated to `width: "80%"`,
`minWidth: 60`, `maxHeight: "85%"`. If the host terminal is below 60
cols, Pi may still mount the overlay (depending on Pi's clipping
policy); the panel detects the narrow width itself in `render(width)`
and emits the hard-block message. This is robust regardless of how
Pi handles minWidth.

Install widget (belowEditor placement) and playback indicator render
unconditionally — they're single-line components that fit any
terminal width down to 20 cols. They're not panel-gated.

## §11 — Pillar F: Help overlay + hotkey routing (REVISED; addresses Codex #8 + Gemini low)

### Hotkey routing rules

1. `?` opens help **only** when no picker is open AND no overlay is
   in front. Inside a picker (search active or not), `?` is a literal
   character forwarded to the search filter.
2. `F1` always opens help, regardless of context — picker search treats
   F1 as escape-equivalent (back out of picker, then open help).
3. `h` is **not** bound — too many accidental triggers (vim users
   pressing `h` in pickers). Documented in help.
4. Help overlay is rendered via `ctx.ui.custom()` so it stacks ABOVE
   any picker/panel — restoring the prior state on close. The previous
   focus, cursor row, search query, and tab are preserved across help
   open/close.
