# pi-listen v7.0.0 — World-class voice extension for Pi CLI

## Mission

Take pi-listen from "STT works, TTS basics work" to "the voice extension users
*expect* a coding agent to ship." End-to-end optimized: pick-once UX, sub-second
TTFB, offline-first by default, telemetry-free, zero footguns.

## Honest baseline (what v6.0.0 actually does today)

| Capability | v6.0.0 status |
|---|---|
| Hold-to-talk STT (Deepgram + 19 local models) | ✅ working, polished |
| `/voice-speak <text>` TTS | ✅ wired but UNTESTED end-to-end with audio |
| Local TTS catalog (14 models) | ✅ catalog data; download triggers on first speak |
| Pick model from settings panel | ❌ read-only display, edit JSON to change |
| Pick voice sid from settings panel | ❌ read-only |
| Pick Deepgram voice from settings panel | ❌ read-only |
| Download with visible progress | ⚠️ implemented in lib, not surfaced in UI |
| Auto-speak agent responses | ❌ feature flag exists, hook not wired |
| Streaming TTS playback | ❌ temp-WAV + spawn (~50-100ms extra latency) |
| First-run onboarding for TTS | ❌ skipped |
| Diagnostic command (`/voice-speak-info`) | ❌ missing |

The user has explicitly asked for: "toggle to select local voice models" and
"option to automatically download those local models." Both are dead-on
v7 territory.

## Scope — what world-class means here

### Pillar A: Discoverability and one-shot setup
1. **First-run TTS onboarding**: when user enables TTS for the first time
   (`/voice-speak-toggle`, settings panel toggle, or `/voice-speak` with
   `ttsEnabled === false`), show a 3-step picker:
   - Backend (Local default, Deepgram alternative)
   - Model (Local: Kitten Nano default + per-language Piper grid; Deepgram:
     12 Aura voices)
   - Voice (sid for Local, Aura id for Deepgram)
   On finish, auto-download the model with progress notifications and play
   a 1-second test sample. Total: 30 seconds from "I want TTS" to
   "audio playing."
2. **Settings panel inline pickers**: Speak tab gets two new sub-pickers
   that mirror the existing `lang-picker` UX pattern:
   - **Model picker**: shows full catalog with size, language coverage,
     install status (✓ installed / ⬇ download on select), recommended
     badge for the device's fitness tier. Search-as-you-type filter.
     Activating a not-yet-installed entry auto-downloads with a progress
     line in `notify()`, then activates.
   - **Voice picker**: lists every voice the current model supports with
     human-readable name + gender + language. For Kokoro multilingual
     v1.0 this is 53 entries; for Piper singletons it's 1 (so we skip the
     picker and inform). For Deepgram, shows the 12 Aura voices filtered
     by `ttsLanguage`.
3. **Speak tab top-row "Status" line**: always-visible single-line summary
   so users instantly see what's currently configured. Format:
   `Local • Kitten Nano v0.2 • voice 0 (Expr-2-M) • 1.0× • EN`
   or `Deepgram • aura-asteria-en • 1.0× • EN`.
4. **`/voice-speak-info` diagnostic command**: prints backend, model,
   voice, language, install status, sample-rate, expected TTFB, audio
   player path, last error (if any). Mirrors the existing `/voice test`
   for STT.
5. **`/voice-speak-models` command**: opens the Speak tab Models picker
   directly (parallel to `/voice-models` for STT). Equivalent to
   `/voice-settings` then ↵ on Model row.

### Pillar B: Auto-download with first-class UX
6. **Auto-download on selection**: when the user picks a not-installed
   model in the picker, immediately call `ensureTtsModelInstalled`. Show
   a notification with byte-count progress (already implemented in
   `tts-local-models.ts`; wire to `cmdCtx.ui.notify` updates every
   ~5% or 1 MB).
7. **Disk-space pre-check**: before download, call existing
   `getFreeDiskSpace`. If `< 2 × model.sizeBytes`, show a clear "you have
   X MB free, model needs Y MB" warning and abort.
8. **Resume on interrupted download**: store partial archives at
   `~/.pi/models/tts/<id>.partial.tar.bz2`; on retry, range-request the
   missing bytes via `Range:` header. Falls back to full re-download if
   the server returns 200 instead of 206.
9. **Download integrity verification**: SHA-256 the extracted directory
   on completion and compare against a known-good hash committed to the
   catalog. Mismatched hashes are surfaced as actionable errors
   ("Download corrupted, retry from /voice-speak-models").
10. **Smart default selection**: on first install with no model picked,
    auto-pick based on `device.systemLocale`:
    - `en` → Kitten Nano (smallest, English-optimal)
    - `zh|ja|ko|es|fr|hi|it|pt` → either matching Piper voice (small) or
      Kokoro multilingual (one-time big install, 9 langs covered)
    - any other → fallback to Kitten Nano + warn that English is the only
      built-in coverage.

### Pillar C: Performance — sub-second TTFB
11. **Streaming local playback**: replace temp-WAV spawn with float→int16
    PCM piped over stdin to `afplay`/`aplay`/`ffplay`. The OfflineTts
    `onProgress` callback already fires per-sentence; route each chunk
    directly to the player's stdin. Cut TTFB from ~700ms (synthesize
    full → write file → spawn) to ~250ms (synthesize first sentence →
    pipe → player starts immediately).
12. **Engine warmup on enable**: when user enables TTS, kick off a
    background `loadSherpa()` + `getOrCreateTts(model, dir)` so the
    first speak isn't blocked by 600ms cold-start init. Cancel if the
    user disables before warmup completes.
13. **Threadpool tuning per model**: extend the `numThreads` logic from
    Parakeet STT to TTS. Kitten Nano scales to 4 threads on M-series;
    Kokoro to 6. Already-validated CPU-only path (CoreML EP regresses
    transformer graphs per sherpa-onnx #2910).
14. **Model preinstall on backend toggle**: when user toggles backend to
    `"local"` and no model is installed, auto-download the default
    Kitten Nano in the background with a progress line. User can keep
    using STT while it pulls.

### Pillar D: Robustness and error UX
15. **Unified error taxonomy**: every TTS error message follows
    `<scope>: <one-line cause>. <one-line fix>.` pattern. E.g.
    `TTS engine: sherpa-onnx unavailable on this platform.`
    `Switch to Deepgram backend in /voice-settings or run on x64/arm64 glibc.`
16. **Graceful Deepgram failure surfacing**: on 401 → show "Invalid
    DEEPGRAM_API_KEY. Update via /voice-settings or DEEPGRAM_API_KEY
    env var." On 429 → "Deepgram rate limited; retry in N seconds."
    On 5xx → "Deepgram service issue; switch to Local backend in
    /voice-settings as fallback."
17. **Mid-flight model deletion**: deleting a model from Downloaded tab
    while it's the active TTS model now safely re-defaults to Kitten
    Nano (already-validated STT pattern, generalize to TTS).
18. **Network airplane-mode test**: dedicated test in CI that disables
    network mid-synthesis and confirms local backend completes the
    chunk it had already started; never makes a fresh fetch.
19. **Single-instance-at-a-time concurrency**: the chain serialization
    we already ship is correct; add a regression test that two concurrent
    `/voice-speak` calls produce non-overlapping audio.

### Pillar E: Auto-speak (the long-promised v6.1 feature)
20. **`after_assistant_message` hook**: when `ttsAutoSpeak === true`,
    pi-listen subscribes to assistant-turn-complete events and pipes
    the message through `speak()`. Hooks into pi-coding-agent's
    extension API.
21. **Smart text filtering for auto-speak**: strip code blocks, ANSI
    escape sequences, markdown link URLs (keep link text), and excessive
    whitespace before synthesis. Code blocks are a hard skip — reading
    `\`\`\`typescript const foo = "bar";\`\`\`` aloud is unhelpful and
    long-form. Surface the skip with a brief tone or status hint.
22. **Auto-speak abort on next user message**: if the user starts typing
    or sending a new message while the previous response is being read,
    immediately abort the in-flight speak. Reuses the existing
    `AbortController` plumbing.
23. **Per-message length cap**: skip auto-speak for messages > 2000
    characters with a status note "(response too long for auto-speak;
    use /voice-speak <selection> to read a portion)". Avoids the
    "agent dumps a 500-line scaffold and we read for 5 minutes"
    failure mode.
24. **Auto-speak rate limit**: max one auto-speak per ~3 seconds. If
    the agent emits multiple rapid messages, queue or drop with a
    visible status indicator.

### Pillar F: Localization and accessibility
25. **All TTS-emitted text uses BCP-47 normalization**: `pt-BR` and
    `pt-br` and `pt_BR` all canonicalize to `pt-BR` at config-load time.
    Already partially implemented in `modelSupportsLanguage`; promote to
    a single `normalizeBCP47` helper used everywhere.
26. **Voice picker shows native-script names where available**:
    `中文` next to `Chinese`, `日本語` next to `Japanese`, etc. for
    Kokoro multilingual voices. Improves discoverability for non-English
    users.
27. **Speed control respects user accessibility preferences**: if
    macOS `defaults read NSGlobalDomain AppleScrollerPagingBehavior`
    or equivalent reduce-motion / accessibility flags are set, default
    `ttsSpeed` to 0.85 (slower for screen-reader-style use cases). Don't
    override an explicitly-set value.

### Pillar G: Tests and gates
28. **130 → 200+ tests**: add coverage for the picker state machine,
    download progress reporting, BCP-47 normalization edge cases, the
    auto-speak text filter (code-block strip), warmup cancellation, and
    smart-default selection.
29. **Real-pi integration test in CI**: stand up a minimal Pi instance
    in CI (using the SDK harness) and run `/voice-speak hello` to
    confirm extension load → command dispatch → audio bytes generated
    (we redirect afplay to /dev/null and verify the WAV bytes match a
    fixture).
30. **Codex review + Gemini bug hunt + godspeed gate** at every step.
    Same Triad pattern that produced the 130/130 v6.0.0 ship.

### Pillar H: Documentation
31. **README rewrite**: lead with TTS examples for v7. Comparison table
    Local vs Deepgram (RTF, network, voice count, license). One-liner
    per Pillar above.
32. **`docs/v7.0.0-tts-guide.md`**: full user guide — onboarding flow,
    catalog with sound samples (link to public Hugging Face audio
    demos for each model), troubleshooting matrix.
33. **Inline command help**: every `/voice-speak*` command's `description`
    field is updated to be self-explanatory in `/help` output.

## Risks and mitigations

### R1: Real-pi audio test difficult to automate
The current smoke skips the actual audio pipeline (afplay → speakers).
Mitigation: stub `afplay` with a wrapper that writes the input WAV to
`/tmp/pi-listen-test-output.wav` and exits 0. The integration test
verifies the WAV bytes (header + ~minimum sample count) without
requiring a real audio device. CI runs without speakers.

### R2: Streaming playback breaks on platforms where stdin pipe is fragile
`afplay` on macOS does not actually accept WAV from stdin (verified —
it requires a file path). Mitigation: streaming playback uses
`ffplay -f f32le -ar <sr> -ac 1 -i -` which handles raw float PCM
from stdin reliably on all three platforms. Fall back to file-based
spawn if `ffplay` not present, with a config hint to install it.
Documented in README.

### R3: Auto-speak hook depends on pi-coding-agent extension event
We need `after_assistant_message` (or equivalent) to be exposed in the
pi-coding-agent extension API. Mitigation: confirm the API surface in
pi-coding-agent 0.70.5 typedefs before building Pillar E. If the event
isn't available, document the gap and fall back to a polling pattern
via `pi.getLastAssistantMessage()` if that exists; otherwise defer
auto-speak to v7.1 with a clear ADR commit.

### R4: SHA-256 verification slows downloads for users on slower disks
A 126 MB file SHA-256 takes ~200ms on M-series, ~500ms on a Raspberry
Pi 5. Mitigation: stream the hash during extraction (the `tar -xj` pipe
already reads every byte), so verification cost is amortized into the
extraction time, not added on top.

### R5: Smart default selection could surprise users
If we auto-select `kokoro-int8-multi-lang-v1_0` for a French user,
they're suddenly downloading 126 MB without explicit consent.
Mitigation: smart default *suggests* in the onboarding picker — never
auto-installs. The user always picks explicitly; we just pre-highlight
the recommendation.

### R6: Disk-space pre-check race
Free space at check time may not be free space at finish time if other
processes write concurrently. Mitigation: use the existing `2x model
size` headroom rule, which gives enough margin for typical concurrent
writes. Surface "(estimated)" in the warning text to manage expectations.

## Versioning

**v7.0.0** — major bump because:
- New first-run TTS onboarding flow (UX surface change)
- New `after_assistant_message` extension subscription (pi-mono dep)

- Description in package.json expands again

Existing v6.x configs load with all new fields defaulted; no migration
breaking changes.

## Definition of done for v7.0.0

1. `pi install npm:@codexstar/pi-listen` then `/voice-speak hello world`
   plays audio on first try without editing JSON. Onboarding handles
   model selection and download.
2. `/voice-settings` Speak tab has working Model picker with download
   progress and Voice picker with named voices.
3. `ttsAutoSpeak: true` causes the agent's responses to be spoken
   automatically with code-block stripping and length cap.
4. Streaming playback measured at < 350ms TTFB on M-series for a 50-word
   response (vs ~700ms in v6).
5. Network disabled mid-session: local backend keeps working with
   already-installed models. Surfaced gracefully.
6. 200+ tests passing, all godspeed gates PASS or VETO=0.
7. README rewrites lead with TTS; v7 user guide complete.
8. npm published, post-publish smoke confirms version + module surface
   + real-pi RPC load.
9. End-to-end audio test run in this session — `/voice-speak-test`
   produces audible output on the user's machine before declaring done.
   No more "shipped but not heard" gap.
10. CHANGELOG documents every Pillar with concrete examples.

## Out of scope for v7.0.0 (deferred to v7.1+)

- Voice cloning (sherpa-onnx Pocket TTS): different use case, niche,
  would dilute the v7 release narrative
- Custom voice training UI: out of scope
- Multi-modal (vision-driven captioning): unrelated
- WebSocket streaming for Deepgram: keep behind `ttsDeepgramStreaming`
  flag; ship to GA in v7.1 once REST is rock-solid in the field
- Mobile / web equivalents: pi-listen is a Pi CLI extension, not a
  cross-platform package
